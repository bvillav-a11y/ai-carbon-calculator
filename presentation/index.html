The logic, the bases, and the deployment of a footprint estimator.

Title. One line: "This is a tour of a small tool that estimates the monthly carbon footprint of personal AI use — and more interestingly, the reasoning and the deployment story behind it." Set the frame: three parts — the choices, the bases (the math + data), and getting it live on gizmos. ~30s.
the problem

Nobody actually knows their number

  • Providers don't disclose per-query energy, water, or training amortisation.
  • "AI is bad for the environment" is a vibe, not a number you can act on.
  • Goal: a defensible estimate from inputs a normal person can answer — and show the working.

Live: ai-carbon-footprint.telus.gizmos.run · public mirror: bvillav-a11y.github.io/ai-carbon-calculator

The motivation. Emphasise the gap: people feel guilty or dismissive, neither is grounded. The bar we set ourselves: defensible, transparent, and driven by answers a non-expert can give. Mention it's live on gizmos internally and mirrored on GitHub Pages. ~45s.
Pillar I

The logic of the choices

Transition. "First, the product reasoning — why the tool is shaped the way it is." ~10s.
the mental model

One pipeline, four stages

survey answers→ inferred parameters→ carbon math→ tangible equivalents

Everything downstream is a pure function of the inferred parameters, so any value can be overridden and the whole chain re-renders live.

This is the spine of the whole app — keep coming back to it. Plain-language answers become physical parameters; parameters feed one math function; the kg result is translated into things people feel. The key engineering property: it's a one-way pure pipeline, so "what-if" is free — change an input, everything recomputes. ~50s.
choice · inputs

A survey, not a parameter form

you ask
"How complex are your tasks?"
"Cloud or on-prem?"
"Interactive or batch?"
you infer
task complexity → model tier
hardware type → PUE
interactivity + frequency → utilisation

Nobody knows their PUE. Everybody knows whether they run batch jobs or chat.

The core product bet: meet people where they are. Asking for PUE or utilisation directly would lose everyone. So we map answerable questions onto the physical parameters. Note the deliberate decoupling: task complexity drives the model tier; interactivity feeds utilisation, NOT the tier. Each inference also stores a reason string, surfaced as a tooltip on the matching control. ~55s.
choice · layout

Outcome first, knobs last

  • Results — the number + tangible equivalents, immediately.
  • Insights & actions — right-size your model, cleaner grid, concrete cuts.
  • More information — parameters, calculation trace, confidence, sources — all collapsed.

Most people want the answer and what to do about it. Experts can open every knob.

Progressive disclosure. The old design led with a parameter form; we flipped it. Land on the outcome, then "here's what you can do," then — for the curious — the full machinery behind four collapsibles under a "More information" header. This is also where engagement telemetry lives: we record which of those menus people actually open. ~50s.
choice · trust

Credibility is a feature

every assumption
editable and sourced — with a confidence bar per factor.
honest by default
"This is probably an underestimate" — we say what's excluded.

Reserved impact-colour (neutral scenario tiers) · WCAG 2.1 AA · ARIA radio-groups + live regions.

For a sceptical technical audience, trust is the product. Three moves: (1) every number is both editable and cited, with an explicit confidence level; (2) we proactively state the estimate is conservative and list what's left out; (3) accessibility is maintained, not bolted on — ARIA patterns, keyboard nav, AA contrast (several palette colours were lifted specifically to pass AA). ~55s.
Pillar II

The bases: the math & the data

Transition into the rigour. "Now the part a technical audience actually wants — what's under the number." ~10s.
computeCarbon()

The chain, as a ledger

raw energy = inTok·in_kWh/kTok + outTok·out_kWh/kTokkWh
÷ utilisationkWh
× PUEkWh
× grid g/kWhoperational kg
+ embodied GPU manufacture, amortisedkg
= total kg CO₂total

One pure function. The on-screen "calculation trace" renders exactly these rows.

Walk each line slowly — this is the meatiest slide. Energy starts per-token, split input vs output. Divide by utilisation (idle GPUs still draw power). Multiply by PUE (data-centre overhead). Multiply by grid intensity to turn kWh into kg. Add amortised embodied carbon. The on-screen ledger in the app mirrors this row for row — the math is never hidden. ~90s; this slide can run long, that's fine.
MODEL_SPECS · kWh per 1k tokens

Output costs ~5× input

Claude Haikuin 0.000090 · out 0.000450
Claude Sonnetin 0.000390 · out 0.001950
Claude Opusin 0.000780 · out 0.003900

■ input — cheap, parallel · ■ output — generated one token at a time, ~5× costlier

Source: Epoch AI — "How much energy does ChatGPT use?" (Sonnet ≈ GPT-4o class).

Two points. First, energy is per-token and asymmetric — output is autoregressive (one token at a time) so it's roughly 5× input; that's why "ask for concise answers" is a real lever. Second, tiers scale ~linearly Haiku→Sonnet→Opus, which is what makes "right-size your model" a genuine carbon decision, not a rounding error. Numbers are grounded in Epoch AI's public estimates. ~60s.
GPU_SPECS · PUE · utilisation

Where the watts leak

gpu
H100 700W · A100 400W (SXM4) / 300W (PCIe) · embodied 130–225 kg CO₂ per card
pue
1.2 hyperscaler · 1.4 typical cloud · 2.0+ on-prem — multiplies IT energy by overhead
utilisation
energy per useful token scales as 1 ÷ utilisation — idle silicon still burns
Three multipliers people never see. GPU choice sets the power draw and carries embodied carbon. PUE is the tax for cooling and networking — 1.4 means 40% on top. Utilisation is the sneaky one: a half-idle GPU doubles the energy attributed to each real token, which is exactly why interactive use is dirtier per token than batch. ~60s.
GRID_MAP · g CO₂ / kWh

The grid often decides it

🇨🇦 Canada BC (hydro)15
🇫🇷 France (nuclear)60
🇧🇷 Brazil136
🇺🇸 US average400
🇵🇱 Poland (coal)700

~47× spread, end to end. Insight fires when grid > 150: "a cleaner grid (~50) would cut X%."

The single biggest swing in the whole model. Same tokens, same hardware: a BC-hydro grid versus Polish coal differ by ~47×. So the app's biggest lever isn't the model — it's where the data centre is. That's why one of the personalised insights is grid-based: above 150 g we compute the saving against a ~50 g hydro/nuclear reference. ~55s.
embodied

The carbon you already spent

embodied_kg = gpuCo2 / (gpuTok · gpuLife · 12) · tokens
// e.g. 150 kg / (50M · 4yr · 12mo) · your tokens

Manufacturing CO₂ amortised over the GPU's lifetime token throughput. Small next to operational — but counted, because honesty.

Most footprint tools ignore embodied carbon entirely. We amortise the chip's manufacturing emissions across all the tokens it will ever serve, then attribute your slice. It's usually small versus operational energy — but including it is part of being defensible rather than convenient, and it's adjustable in the advanced panel. ~45s.
tangible equivalents

From kg to something you feel

6 km
driven per kg CO₂
22 kg
absorbed / tree / year
$50
social cost / tonne

Each equivalent gets its own accent colour — deliberately distinct from every semantic colour in the math.

A kilogram of CO₂ is meaningless to most people. So we translate: kilometres driven, trees-years to offset, dollars of social cost. Note the colour discipline — these three accents (orange/teal/pink) are reserved exclusively for equivalents, never reused for a parameter, so the eye learns "this colour = a real-world analogy." ~45s.
confidence & what's excluded

Say what you don't know

Energy per tokenmed
PUE (inferred, undisclosed)low–med
Grid intensityhigh

Excluded: training amortisation · networking · cooling water · non-GPU embodied. computeCarbon is one pure function — the scenario tiers just call it with a different model.

Close the rigour section on humility. Each factor carries a confidence level — grid is well-measured, PUE is inferred and uncertain. We list the big exclusions explicitly so nobody thinks the number is complete. And the engineering punchline: because computeCarbon is a single pure function, the Haiku/Sonnet/Opus scenario comparison is literally the same function called three times — no duplicated math to drift. ~55s.
Pillar III

The process of deployment

Transition to the war-stories. "Now how it actually got live — and everything that fought back." ~10s.
where it lives

Two targets, one source file

GitHub Pages
public demo · serves index.html directly · /collect 404s, collection no-ops
TELUS gizmos
the real home · same Worker serves the app and a same-origin POST /collect → per-app D1

The entire calculator is one index.html — HTML + one CSS block + one vanilla JS block. No framework.

Set up the deployment shape. The calculator is a single no-framework index.html, which makes the public mirror trivial. The interesting target is gizmos — TELUS's internal app platform — where the same file is served by a Worker that also collects anonymous survey data into a per-app SQLite (D1) database. The constraint that everything is one HTML file drives the whole embed story coming up. ~50s.
gizmos · architecture

One plain Worker does everything

GET  /         → serve base64-embedded index.html
POST /collect  → upsert survey row into D1 (by sessionId)
GET  /export   → CSV dump of responses

Per-app D1 auto-provisioned · table created lazily (CREATE TABLE IF NOT EXISTS) · consent-gated.

The shipped architecture, deliberately minimal. One Worker, three routes. The HTML is embedded in the Worker as a base64 string. Collection is a same-origin POST, upserting by session id so re-runs update one row. Export is a CSV endpoint we use to verify data, because the platform's DB CLI needs a browser session. Now — why does it look like this? Because three things broke first. ~50s.
what fought back

Three platform lessons

1 · module load
a bare import 'hono' → 500 on load, empty logs → rewrite as a plain Workers module, no imports
2 · reserved path
/api/* is platform-reserved → app POSTs silently blocked → move to /collect & /export
3 · cross-origin auth
Apps-Script POSTs dropped the auth cookie (3rd-party-cookie stripping) → pivot to same-origin collection
The fun slide — tell these as stories. (1) The deploy 500'd with empty logs; bisection showed the loader won't resolve a bare hono import at runtime, so any framework worker dies on load — fix was a plain Workers module with zero imports. (2) Collection POSTs vanished; turns out /api/* is reserved by the platform and app POSTs to it are blocked — renaming to /collect fixed it instantly. (3) The original plan POSTed to a Google Apps Script, but background cross-origin requests get their auth cookie stripped, so it silently 401'd — which is the whole reason collection became same-origin. Each failure shaped the architecture on the last two slides. ~90s.
build · embed.mjs

One source file, regenerated

index.html  +  worker.template.ts
      │  node scripts/embed.mjs   // base64-inline the HTML
      ▼
gizmos-app/src/index.ts  →  gizmos push --org telus

Base64, not a template literal — the inline <script> is full of backticks and ${...}. Regenerate after every edit.

How a change ships. index.html stays the single source of truth; a small build step base64-encodes it into the Worker template, producing the deployable file we push. Why base64 and not a JS template string? The app's inline script is full of backticks and dollar-brace — it would shred a template literal. The one discipline: re-run embed after any HTML change, or you deploy stale. ~45s.
collection · telemetry

Anonymous, consent-gated, one row per visit

  • Upsert by sessionId; baseline frozen at first results render.
  • Engagement: dwell_ms, which menus opened, share, retake.
  • Schema evolves via best-effort ALTER TABLE … ADD COLUMN on an existing table.
What we learn and how. Only anonymous signals — the survey answers, the computed number, and engagement: how long on results, which "More information" menus they opened, whether they shared or retook. All consent-gated, one upserted row per session. The schema note matters for the next slide's lesson: the prod table predates the telemetry columns, so the Worker runs idempotent ALTER TABLE migrations to add them — CREATE IF NOT EXISTS won't touch an existing table. ~55s.
what to steal

Four things that generalise

  • Platform constraints shape architecture — the plain Worker + same-origin collection are scars, not preferences.
  • Ship the schema and its migration together — CREATE IF NOT EXISTS never alters a live table.
  • Transparency is a feature — editable, sourced, confidence-rated beats a confident black box.
  • Right-size your model — the app's own advice, applied to how we build.
The portable lessons. One: don't fight the platform — its limits became the design. Two: the migration lesson is the one I'd tattoo — a schema change isn't done until its migration ships with it. Three: for a credibility tool, showing the working IS the product. Four: meta-point — the tool tells users to pick the smallest model that does the job; same applies to our own engineering. ~55s.
